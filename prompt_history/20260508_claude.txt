## okamoさんの提案（Cognito API で sub → custom:xxx を取得）への意見

**提案の方向性は正しく、むしろ長期的にはこちらを採用すべきです。** 理由：

| 観点 | IDトークンのclaimに頼る方式 | Cognito API lookup方式 |
|------|---------------------------|----------------------|
| 耐改ざん性 | JWT署名で保証（ただしトークン有効期間中の古いデータ） | sub検証後にサーバー側で最新値を取得 = 常に正 |
| ロール変更の即時反映 | トークン再発行まで古いroleが使われる危険 | 即時反映される |
| 実装の複雑さ | シンプル | Cognito API呼び出し + キャッシュが必要 |

`sub` はJWT RS256署名で改ざん不能であり、そのsubに紐づくCognito属性をサーバー側で取得する方式は**セキュリティモデルとして健全**です。

## 実装上の注意点（リクエストごとのAPI呼び出しを避ける）

**懸念点**: `AdminGetUser` をリクエストごとに毎回呼ぶと100〜200msの追加レイテンシーが発生します。

**解決策**: `sub` をキーとした短期インメモリキャッシュ（60秒TTL）を追加します。Lambda コンテナ内のモジュールレベル変数はウォームスタート間で保持されるため、有効です。

また、`AdminGetUser` の `Username` パラメータには JWT に含まれる `cognito:username`（メールアドレス）を使えます：

```typescript
// src/lib/admin-auth.ts に追加するキャッシュ付きlookup
const ATTR_CACHE = new Map<string, { role?: string; siteIds?: string[]; exp: number }>();

async function lookupCognitoAttributes(
  sub: string,
  cognitoUsername: string  // JWTの cognito:username クレーム
): Promise<{ role?: string; siteIds?: string[] }> {
  const hit = ATTR_CACHE.get(sub);
  if (hit && hit.exp > Date.now()) return hit;

  const { CognitoIdentityProviderClient, AdminGetUserCommand } =
    await import('@aws-sdk/client-cognito-identity-provider');
  const client = new CognitoIdentityProviderClient({
    region: process.env.COGNITO_USER_POOL_ID!.split('_')[0],
  });
  const resp = await client.send(new AdminGetUserCommand({
    UserPoolId: process.env.COGNITO_USER_POOL_ID!,
    Username: cognitoUsername,
  }));

  const attrs = Object.fromEntries(
    (resp.UserAttributes ?? []).map(a => [a.Name, a.Value])
  );
  const result = {
    role: attrs['custom:role'],
    siteIds: (() => {
      try { return attrs['custom:siteIds'] ? JSON.parse(attrs['custom:siteIds']) : undefined; }
      catch { return undefined; }
    })(),
    exp: Date.now() + 60_000, // 60秒キャッシュ
  };
  ATTR_CACHE.set(sub, result);
  return result;
}
```
